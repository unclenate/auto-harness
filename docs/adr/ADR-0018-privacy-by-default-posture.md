# ADR-0018: Privacy-by-Default Posture

**Status:** Accepted
**Date:** 2026-06-03
**Author:** @unclenate
**Reviewers:** @unclenate
**Context sources:**

- `docs/operating-principles.md` § 11 — the canonical principle this ADR
  makes operational
- `docs/superpowers/specs/2026-06-03-privacy-by-design-design.md` — the
  full design spec: module structure, artifact schema, validator contract,
  bootstrap integration, and forcing-artifact list

## Context

Privacy by design is a cross-cutting concern: it touches every consumer
project that handles personal or sensitive data, and the harness is
explicitly chartered to *help* consumers implement it by default — not
merely permit it. The obligation is proactive, not reactive.

The framework already carries a mature pattern for cross-cutting,
jurisdiction-sensitive domain concerns: the deep-domain
**jurisdiction-neutral-core / forcing-artifact / bias-guardrail** triad,
first applied in `domains/healthcare-*` (PRD-0017). That pattern separates
the universal floor (jurisdiction-neutral principles) from the
consumer-declared legal regime (GDPR, CCPA/CPRA, LGPD, PIPEDA, PIPL, …),
preventing the harness from assuming a jurisdiction it has no grounds to
assume.

Privacy by design is the first *cross-vertical* application of that same
triad. The content spine is Cavoukian's seven Foundational Principles of
Privacy by Design — jurisdiction-neutral by design — with the applicable
legal regime declared by the consumer at initialization time, never
hard-coded by the harness.

Three options were evaluated against the requirement that the harness
genuinely default to privacy, not merely tolerate it:

1. **Opt-in overlay only** — consumer adds `management/privacy-by-design`
   when they want it.
2. **Default-on, opt-out** — the bootstrap-generated manifest includes the
   overlay; a documented exemption removes it.
3. **Kernel-mandatory, no opt-out** — the overlay is non-removable,
   equivalent in enforcement weight to the kernel itself.

The design spec cited above evaluates all three against the
`management/privacy-by-design` module's target validator posture (WARN),
the realistic population of consumer projects (includes data-free libraries
and pure-compute tools), and the framework's operating principle on
consumer autonomy. Operating-principle § 11 records the outcome as
governing doctrine; this ADR records the architectural decision and the
rejected alternatives.

## Decision

Adopt **default-on, opt-out** for `management/privacy-by-design`.

Every manifest generated by `harness bootstrap` includes
`management/privacy-by-design` in its active module list. A project with
genuinely no personal or sensitive data may opt out — but opt-out is
explicit and recorded (a `none` exemption in `docs/privacy/privacy-profile.md`),
never silent. The exemption documents *why* no privacy coverage is needed;
an empty or missing profile is itself a validator warning.

The validator posture is **WARN** (module-gated): the validator runs only
when the overlay is active, issues advisory warnings on privacy-risk
patterns, and does not block green CI on the harness's own self-dogfood
build while the module is being delivered (ship-as-catalog). Companion
rules enforce that privacy artifacts update when data-handling paths
change. Human review gates prevent risky merges.

PRD-0018 specifies the full build: module scaffolding, artifact schema,
validator rules, bootstrap wiring, and the `none`-exemption path.

## Alternatives considered

**Opt-in overlay only.** The consumer adds `management/privacy-by-design`
when they decide they need it. Rejected: "by default" in the principle
name is not decoration. An opt-in posture turns privacy coverage into a
deliberate affirmative choice; most projects that *should* activate it
never will because the cost of inaction is invisible at bootstrap time.
Opt-in fails the proactive requirement of Cavoukian Principle 1 at the
harness layer.

**Kernel-mandatory (no opt-out).** The overlay is a non-removable
constituent of the kernel, equivalent in enforcement weight to
`platform/core/kernel/`. Rejected: this imposes the full privacy-artifact
ceremony on genuinely data-free projects — static libraries, pure-compute
tools, CLI wrappers — where the ceremony produces no safety benefit and
only compliance drag. The harness's consumer-autonomy principle (§ 3)
explicitly reserves opt-out for cases where a module would produce dead
weight. Default-on / opt-out preserves the protection for the vast
majority while giving data-free projects a documented, non-silent escape.

## Consequences

**Positive:**

- Every bootstrapped project activates privacy-by-design coverage
  without requiring the consumer to remember to opt in. The Cavoukian
  "privacy as the default setting" principle is met at the harness layer,
  not left to consumer initiative.
- The `none`-exemption path gives data-free projects a clean, documented
  exit that is itself auditable — an exemption file that explains why
  no coverage is needed is more informative than a missing file.
- The jurisdiction-neutral-core / forcing-artifact / bias-guardrail
  pattern gains its first cross-vertical application. The pattern is now
  reusable for finance, logistics, and any future domain that separates
  universal principles from regime-specific rules — exactly the
  deep-domains framework goal.
- The WARN posture keeps auto-harness's own CI green while the module
  ships as catalog (self-dogfood deferred, validator is module-gated).
  No cargo-cult CI noise is introduced.

**Negative / costs:**

- Bootstrap-generated manifests now include one additional active module.
  Projects that opt out must write and maintain the `none`-exemption
  entry in `docs/privacy/privacy-profile.md`. This is a one-time
  low-effort cost, but it is a new obligation that did not exist before.
- The WARN posture means the validator is advisory, not blocking.
  Privacy *outcomes* remain Asserted-only (human-judged) per § 10;
  artifact presence and companion-rule pairing are Enforced. This is a deliberate limitation,
  not an oversight.

**Watch:**

- If a consumer project populates a `none` exemption and later adds
  data-handling code, the validator should detect the mismatch and
  re-prompt. PRD-0018 specifies this as a required validator rule;
  confirm it is implemented before marking the module production-ready.
- The bias-guardrail pattern (preventing the harness from assuming a
  legal regime) must be tested against all bootstrap code paths that
  touch `docs/privacy/privacy-profile.md`. A bootstrap that silently
  writes a default regime would contradict the jurisdiction-neutral
  intent.
